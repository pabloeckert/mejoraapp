## Cambios realizados

### 1. Eliminado código muerto `ai.ts`
- **Archivo:** `src/services/ai.ts` → ELIMINADO
- **Razón:** Servicio de IA client-side con ofuscación XOR. Nadie lo importaba. Las IAs ya funcionan 100% server-side via Edge Functions (`moderate-post`, `moderate-comment`, `generate-content`).
- **Impacto:** -200 líneas de código muerto. Elimina superficie de ataque (API keys en localStorage).

### 2. Reemplazado "puntito secreto" por botón admin legítimo
- **Archivo:** `src/pages/Auth.tsx`
- **Antes:** Un puntito de 2x2px invisible debajo del logo
- **Ahora:** Botón con icono Shield + texto "Admin" que se expande a "Modo admin" al activarse
- **Razón:** El "puntito" era security by obscurity — cualquiera que lee el código lo sabe. Mejor ser transparente: el acceso admin requiere credenciales válidas, no oscuridad.

### 3. Edge Function `verify-admin` creada
- **Archivo:** `supabase/functions/verify-admin/index.ts`
- **Función:** Verifica el rol admin server-side usando `SUPABASE_SERVICE_ROLE_KEY`
- **Uso:** Llamada desde `AdminLoginForm.tsx` y `Admin.tsx` para verificar permisos
- **Ventaja:** La verificación de rol ya no depende del cliente (RLS). Un usuario no puede inyectar una respuesta falsa.

### 4. AdminLoginForm usa verificación server-side
- **Archivo:** `src/components/auth/AdminLoginForm.tsx`
- **Cambio:** Ahora llama a `supabase.functions.invoke("verify-admin")` en lugar de hacer query directo a `user_roles`
- **Ventaja:** La verificación de rol usa service_role key, no anon key. Más seguro.

### 5. Admin.tsx usa verificación server-side
- **Archivo:** `src/pages/Admin.tsx`
- **Cambio:** En lugar de query directo a `user_roles`, ahora invoca `verify-admin` Edge Function
- **Ventaja:** Misma razón que #4

### 6. Creado `.env.example`
- **Archivo:** `.env.example`
- **Propósito:** Documentar variables de entorno requeridas sin exponer valores reales

### 7. Script SQL de migración de seguridad
